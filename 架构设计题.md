# 架构设计题：

## **模块一：Context / Performance**

### 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

长会话上下文膨胀的核心矛盾，在于有限的上下文窗口与对话信息完整性、交互连贯性之间的平衡。解决思路并非单纯截断或压缩，而是基于信息的价值密度与时效性分层治理，在控制 Token 总量的前提下，最大化保留对当前交互有效的信息。

#### 一、分层递进的压缩策略

1. **基础层：滑动窗口 + 永久锚点机制**

   保留最近 6-10 轮的完整原始对话，该区间与当前话题关联度最高，不做任何压缩处理，保障近场交互的绝对连贯。将系统指令、核心任务目标、用户明确的硬性约束（技术栈、输出规范、禁忌规则等）设为永久锚点，固定置于上下文头部，不参与压缩与截断，作为全对话的基准信息。超出滑动窗口的历史对话，进入下一层级的压缩流程，而非直接丢弃。

2. **中间层：结构化生成式摘要**

   对窗口外的对话历史，通过轻量模型生成结构化摘要，而非泛化的自然语言概括。摘要固定覆盖五个维度：对话核心议题与演进主线、已达成的关键决策与共识、用户明确的偏好与约束、工具调用记录与核心结果、待跟进的遗留事项。

   压缩采用渐进式执行：当上下文占用达到窗口阈值的 70% 时，仅将最早的 2-4 轮对话合并入摘要，避免一次性批量压缩造成的信息断层。

3. **储备层：语义检索式历史库**

   更早的全量对话历史不直接占用上下文窗口，经向量化后存入对话专属向量库，转为 “按需召回” 模式。每轮交互时，基于当前用户 Query 召回 Top-K 最相关的历史片段，与近期对话、结构化摘要拼接为最终生效上下文。该模式将 “全量携带” 转为 “按需取用”，大幅降低长会话的 Token 基线。

4. **优化层：结构化实体沉淀**

   将对话中产生的确定性实体信息（环境参数、项目路径、版本信息、业务数据等）提取为结构化键值对，注入系统提示的固定字段，无需在对话历史中重复留存，进一步释放对话历史的空间。

#### 二、压缩后的对话连贯性保障

连贯性的核心是 “用户感知不到信息丢失”，主要通过四类机制保障：

- **近场信息完整性**：近期对话完整保留，确保用户提及上几轮内容时，响应无任何偏差。
- **高价值信息强保护**：将用户约束、核心决策、工具结果、报错信息标记为不可压缩项，摘要中必须完整留存，禁止因压缩丢失关键信息。
- **一致性校验机制**：每次摘要更新后，通过轻量模型做关键信息自检，确认核心目标、关键实体、核心约束无缺失，缺失项自动补全。
- **话题对齐召回**：每轮交互基于当前意图定向召回相关历史，话题切换时自动过滤无关内容，既控制 Token 规模，也保证当前话题内的信息密度，避免无关信息干扰对话逻辑。

## **模块二：Memory**

###  和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

用户跨周期重复提问时，记忆召回的核心目标是 “精准、不突兀、不失效”，避免 “全量检索导致的噪声干扰” 与 “检索缺失导致的失忆感”。合理的召回体系需建立在分层存储的基础上，通过多阶段筛选实现精准匹配。

#### 一、前置基础：分层记忆存储架构

按记忆的生命周期与使用频率，将记忆分为三层存储：

- 工作记忆：当前会话的实时上下文，直接参与模型推理。
- 短期记忆：近 3 天的对话记录，留存结构化摘要与关键对话片段。
- 长期记忆：超过 3 天的历史对话，以 “向量索引 + 结构化标签 + 原始片段” 三态存储，支撑多维度召回。

#### 二、全流程召回逻辑

1. **触发判断**

   并非每轮交互都触发召回，仅满足两类条件之一时启动：一是用户输入包含明确回溯关键词（如 “上次”“之前提到的”）；二是当前 Query 与历史问题的语义相似度超过预设触发阈值。无明确召回信号时不执行检索，降低无效算力消耗。

2. **粗排：语义召回**

   通过当前 Query 的向量表示，在长期记忆库中召回 Top20 相关片段，保证召回覆盖率，避免高相关内容漏检。

3. **精排：多维度加权排序**

   对粗排结果从四个维度综合打分，排序后取 Top3-5 注入上下文：

- 语义匹配度：当前问题与历史片段的语义相关性，为核心权重项。
- 时间衰减因子：远期记忆权重低于近期记忆，但高匹配度内容不受时间维度过度压制。
- 重要性权重：核心决策、用户个人信息、业务结论权重高；日常寒暄、过渡性对话权重低。
- 频次因子：用户多次提及或询问过的内容，权重相应提升。

1. 有效性校验

   排序后增加两层校验：一是时效性校验，过滤已过期的时间敏感信息（如临时安排、阶段性状态）；二是语境一致性校验，结合当前对话上下文，排除语义相似但话题场景完全无关的记忆片段。

#### 三、召回后的融合与迭代

- 召回结果以 “历史背景信息” 的形式置于上下文固定区域，不打乱对话时序，仅作为推理背景。
- 回复中自然融合历史结论，避免生硬的 “我记得你之前问过” 式表述；若召回到多个不同语境下的答案，不做主观推断，主动向用户确认对应场景。
- 若本次交互更新了历史结论，同步更新记忆库，将旧版本标记为 “已迭代”，避免后续召回过期结论。

## **模块三：Task** 

### 对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

长程任务执行中目标遗忘的本质，是上下文注意力随执行步骤增加被稀释，进而导致执行路径偏离初始目标。行业内主流的解决方案共六类，各自适配不同场景，存在明确的选型权衡。

**1.目标持久化锚定**

核心原理：将最终目标、验收标准、核心约束写入系统提示词头部，每轮推理全程携带，不参与压缩；进阶实现为每步执行前强制模型复述目标，强化目标对齐。

- 优势：实现成本极低，兼容所有模型，是长程任务的基础标配；目标全程可见，可解决基础的遗忘问题。
- 局限：占用上下文 Token 资源；仅能被动提示，无法主动纠正已发生的偏差；不支持目标动态调整。

**2.任务拆解与里程碑管控**

核心原理：任务启动前先将长程目标拆解为有序子任务与里程碑节点，每个子任务定义明确的输入、输出与验收标准；执行按节点推进，完成一个校验一个，整体进度持久化存储，每轮执行时加载。

- 优势：将大目标拆解为短周期目标，大幅降低目标漂移概率；进度可追踪，异常可早发现，用户体感清晰。
- 局限：前期规划有额外开销，复杂任务的拆解方案若不合理，会导致后续执行卡壳；拆解逻辑偏僵化，不适合高探索性的任务。

3.状态机驱动的工作流编排

核心原理：通过显式状态机定义任务全流程（规划、执行、校验、复盘等节点），状态跳转有明确的触发条件；任务的当前进度、已完成结果、目标参数全部存储于外部状态中，执行时加载对应状态与数据。典型实现如 LangGraph。

- 优势：状态完全显式化，由程序控制流转，从架构层面杜绝目标遗忘，可靠性最高；支持循环、分支、错误重试等复杂逻辑；任务支持中断与恢复。
- 局限：架构复杂度高，开发与调试成本大；预设状态难以覆盖所有意外场景，遇到计划外情况易陷入停滞。

**4.周期性自反思机制**

核心原理：每执行 3-5 步工具调用，强制触发一次反思节点，让模型对照初始目标，自查执行路径是否偏离、是否存在无效步骤，并修正后续执行方案，反思结论更新至任务状态。

- 优势：可主动纠正执行偏差，不仅解决目标遗忘，还能优化执行路径；无需预先拆解任务，适配探索性、不确定性强的长程任务。
- 局限：增加额外模型调用，提升延迟与成本；反思效果高度依赖模型能力，弱模型无法完成有效自查；反思频率需精准权衡，过高拖累效率，过低纠正不及时。

**5.多角色 Agent 协作**

核心原理：拆分执行与监督两个角色，执行 Agent 专注分步工具调用与具体落地，监督 Agent 全程校验目标对齐度，发现偏差立即叫停并纠正。

- 优势：职责分离，监督角色专注目标对齐，容错率高；可并行处理多维度任务。
- 局限：架构复杂度成倍提升，多 Agent 通信与同步成本高；Token 消耗显著增加，小型任务投入产出比过低。

**6.外部任务图谱持久化**

核心原理：将任务目标、子任务、依赖关系、执行结果、核心参数全部结构化存入图数据库，形成任务图谱；每轮执行仅加载当前节点及上下游信息，完全不依赖上下文留存目标。

- 优势：彻底脱离上下文依赖，从根源解决目标遗忘；支持超大规模长程任务，复杂度上限高；支持跨会话、跨端延续同一任务。
- 局限：实现成本最高，需完整的任务图谱数据模型与提取逻辑；结构化提取的准确率直接决定执行效果，提取误差会导致整体路径偏离。

## **模块四：Tool / Session Runtime**

### 如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

会话处于 Busy 状态时的事件处理，核心原则是**单会话串行执行**。对话上下文具有强时序依赖，并行处理必然引发状态竞态，导致上下文混乱。行业通用方案为 “单会话事件队列 + 优先级调度”，兼顾状态一致性与用户体验。

#### 一、基础架构：单会话专属事件队列

每个会话维护独立的事件队列，所有外部事件（用户消息、异步工具回调、超时事件、控制指令）均先进入队列，运行时按优先级、按顺序串行处理。会话状态仅在事件处理周期内变更，保证原子性。

#### 二、事件优先级定义

优先级从高到低分为三级：

1. 控制指令（停止、取消、中断）：最高优先级，可打断当前执行流程。
2. 异步工具完成事件：次高优先级，属于当前活跃任务的组成部分。
3. 用户普通消息：最低优先级，排队等待处理。

#### 三、核心事件的处理逻辑

1. **异步工具完成事件**

   首先通过工具调用 ID 做归属校验，确认属于当前活跃任务；若为已取消任务的回调，直接丢弃。

   校验通过后，将工具结果写入会话状态，唤醒当前推理流程继续执行，直至本轮任务完整结束、会话状态重置为 Idle 后，再处理队列中下一个事件。若工具调用返回异常，按预设策略执行重试或报错流程，结束当前任务后再处理后续事件。

2. **用户新消息到达**

   若为普通业务消息，直接进入队尾，同时立即向用户返回状态回执，告知当前任务正在处理、将尽快响应，避免用户误以为消息丢失；队列中已有待处理消息时，不重复发送回执。

   若为停止、取消类控制指令，触发中断逻辑：终止当前工具调用与推理流程，原子化将会话状态重置为 Idle，再响应用户的控制指令。

   同时设置背压保护机制，队列长度超过阈值时，提示用户稍后发送，避免队列无限堆积拖垮服务。

#### 四、可靠性保障机制

- 状态原子性：会话状态的所有变更为原子操作，避免中间态引发的状态错乱。
- 事件幂等：所有事件携带唯一标识，重复投递仅处理一次。
- 队列持久化：事件队列落盘存储，服务重启后可恢复并继续处理，避免用户消息丢失。

## **模块五：Agent Runtime 架构对比**

### Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

Claude Code 与 OpenAI 兼容式工具调用，本质代表了两种设计路线：前者是**内联流式的灵活范式**，后者是**结构化 API 的标准化范式**。GLM、豆包等国内主流模型均遵循 OpenAI Function Calling 标准，二者在设计逻辑与适用场景上存在显著差异。

### 一、核心设计差异

| 对比维度 | OpenAI 兼容式 Function Calling                               | Claude Code XML 内联式                                       |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 输出形式 | 独立的`tool_calls`结构化字段，与对话内容完全分离             | 内联于自然语言输出流，通过 XML 标签包裹，与思考过程、回复内容混合输出 |
| 参数约束 | 严格遵循预定义的 JSON Schema，格式校验前置                   | 无强制 Schema 约束，标签内内容自由组织，灵活度极高           |
| 调用节奏 | 离散轮次模式：一轮推理→一次工具调用→一轮推理，严格的请求 - 响应结构 | 流式穿插模式：单轮生成流中可连续穿插多个工具调用，边生成边执行 |
| 解析方式 | API 直接返回结构化对象，无需额外解析                         | 需自行实现流式 XML 解析，识别标签边界与内容                  |
| 可观测性 | 推理与工具调用分离，内部思考过程不可见                       | 思考过程与工具调用同步输出，天然具备全链路可观测性           |

### 二、各自的设计优缺点

#### 1. OpenAI 兼容式 Function Calling

优势：

- 接入成本极低，API 直接返回结构化数据，无需自定义解析逻辑，开发效率高。
- 强 Schema 约束下，参数格式错误率低，工业级场景的稳定性与可预期性更强。
- 为行业事实标准，生态成熟度高，主流框架与工具链全面兼容，跨模型迁移成本低。
- 工具调用与对话内容解耦，状态流转逻辑清晰，调试与问题定位成本低。

局限：

- 灵活性不足，所有工具需预先定义完整 Schema，不适配参数复杂、形态多变的场景（如自由代码编辑、开放性检索）。
- 多步任务需多次 API 往返，整体延迟较高。
- 推理与执行割裂，模型无法边思考边验证，不符合自然解题逻辑，复杂长链路任务的表现存在上限。

#### 2. Claude Code XML 内联式

优势：

- 灵活性极强，无 Schema 限制，参数可承载代码、自然语言、结构化数据等任意形态，高度适配编程助手、科研探索等复杂场景。
- 流式穿插调用可减少 API 往返次数，降低整体延迟；支持边生成边执行，交互体验更流畅。
- 推理与执行自然融合，模型可按 “分析 - 验证 - 再推理” 的自然逻辑推进，复杂长程任务的完成率更高。
- 全链路思考过程透明，调试与审计成本低。

局限：

- 接入门槛高，需自行实现流式 XML 解析、标签匹配、异常容错等逻辑，开发工作量显著提升。
- 无强制格式校验，存在标签不闭合、参数格式异常等问题，需要大量异常兜底逻辑。
- 属于厂商专属范式，非行业通用标准，跨模型迁移成本高，第三方生态支持度有限。
- 思考过程、工具调用、用户可见内容混合输出，需精准划分展示边界，处理不当易暴露内部逻辑，破坏用户体验。

OpenAI 兼容式架构更适配标准化、确定性强的业务场景（如业务客服、固定 API 型智能助手），优先保障稳定性与开发效率；Claude Code 的内联式架构更适配复杂探索型 Agent 场景（如编程助手、科研辅助 Agent），优先保障灵活性与任务完成能力。
